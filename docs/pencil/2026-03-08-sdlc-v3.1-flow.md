# SDLC Skills System v3.1 - Flow-Based Design

**Date**: 2026-03-08
**Version**: 3.1 (Flow-Based)
**Status**: Design

## Overview

基于流程的 SDLC 技能系统，强调阶段之间的关联和顺序，支持完整的开发工作流。

## 核心概念

```
┌─────────────────────────────────────────────────────────────────────────────────┐

状态流转:
┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐

                                                           │

└────────┘    └────────┘    └────────┘    └────────┘    └────────┘

  [MERGE]
```

## 阶段定义对比

| 阶段 | 用途 | 检查内容 | 输入 | 输出 |
|-----|------|---------|------|------|
| **validate** | 代码质量 | Lint, 类型检查, 格式 | 代码 | 质量报告 |
| **verify** | 需求验证 | 实现是否符合 spec | 代码 + spec | 验证报告 |
| **test** | 功能测试 | 单元测试, 集成测试 | 代码 | 测试报告 |
| **secure** | 安全检查 | 漏洞扫描, 依赖检查 | 代码 | 安全报告 |
| **cr** | 代码审查 | 代码质量, 最佳实践 | 代码 | 审查意见 |

```
validate + verify + test + secure = 质量保证

validate: 代码写得对不对？
verify:   功能做得对不对？
test:     功能能不能用？
secure:   有没有安全问题？
cr:       代码好不好？
```

## 质量保证流程详解

### 1. validate - 代码质量检查

**目的**: 代码写得对不对？

**检查内容**:
```bash
/sdlc validate
├─ lint       # 代码风格检查
│  ├─ ESLint/Prettier (前端)
│  ├─ golangci-lint (Go)
│  └─ pylint/black (Python)
│
├─ typecheck  # 类型检查
│  ├─ TypeScript
│  ├─ Go types
│  └─ mypy (Python)
│
└─ format     # 代码格式化
   ├─ Prettier
   ├─ gofmt
   └─ black
```

**输出**: 代码质量报告

**示例**:
```
✓ ESLint: 0 errors, 3 warnings
✓ TypeScript: No type errors
✓ Prettier: Formatted 5 files
```

---

### 2. verify - 需求验证

**目的**: 功能做得对不对？

**检查内容**:
```bash
/sdlc verify
├─ spec_coverage     # spec 需求覆盖率
│  └─ 检查所有 spec 中的需求是否已实现
│
├─ implementation     # 实现验证
│  ├─ API 接口是否符合 spec
│  ├─ 数据结构是否正确
│  └─ 业务逻辑是否完整
│
└─ design_match      # 设计一致性
   ├─ UI 是否符合设计图
   ├─ 组件结构是否合理
   └─ 交互逻辑是否正确
```

**依赖**: 读取 `./docs/spec/{date}-*.md`

**输出**: 需求验证报告

**示例**:

## 技能详细定义

### `/sdlc validate`

**描述**: 代码质量检查

**用法**:
```bash
/sdlc validate [check]

checks:
  lint       - Lint 检查
  typecheck  - 类型检查
  format     - 格式化
  all        - 全部检查 (默认)
```

**完成条件**:
- [ ] Lint 通过
- [ ] 类型检查通过
- [ ] 代码已格式化

---

### `/sdlc verify`

**描述**: 验证实现是否符合 spec 需求

**用法**:
```bash
/sdlc verify [spec_file]

# 如果不指定 spec_file，自动查找最新的 spec 文档
```

**检查内容**:
1. **需求覆盖**: spec 中的所有需求是否已实现
2. **API 匹配**: 接口定义是否符合 spec
3. **数据结构**: 数据模型是否正确
4. **业务逻辑**: 业务逻辑是否完整
5. **设计一致性**: 实现是否符合设计

**依赖**: `doc`, `cache`

**输出**: `./docs/verify/{date}-{title}.md`

**完成条件**:
- [ ] 所有需求已实现
- [ ] API 接口匹配
- [ ] 数据结构正确
- [ ] 业务逻辑完整

---

### `/sdlc test`

**描述**: 功能测试

**用法**:
```bash
/sdlc test [type]

types:
  unit      - 单元测试
  integ     - 集成测试
  e2e       - 端到端测试
  coverage  - 覆盖率分析
  all       - 全部测试 (默认)
```

**完成条件**:
- [ ] 单元测试通过
- [ ] 覆盖率达标
- [ ] 无失败测试

---

### `/sdlc secure`

**描述**: 安全检查

**用法**:
```bash
/sdlc secure [check]

checks:
  vuln       - 漏洞扫描
  deps       - 依赖检查
  secrets    - 密钥检测
  all        - 全部检查 (默认)
```

**完成条件**:
- [ ] 无安全漏洞
- [ ] 依赖安全
- [ ] 无密钥泄露

---

### `/sdlc cr`

**描述**: 代码审查

**用法**:
```bash
/sdlc cr [scope]

scope:
  code       - 代码审查
  arch       - 架构审查
  all        - 全面审查 (默认)
```

**完成条件**:
- [ ] 审查完成
- [ ] 问题已修复
- [ ] 获得 LGTM

## 使用示例

### 完整流程

```bash
# 1. 开始
/sdlc start feature "用户认证"
# → research → spec → coding

# 2. 编码完成后，质量保证流程
/sdlc next
# → validate (代码质量)
/sdlc next
# → verify (验证符合 spec)
/sdlc next
# → test (功能测试)
/sdlc next
# → secure (安全检查)
/sdlc next
# → cr (代码审查)

# 3. 提交
/sdlc next
# → commit → pr → MERGE
```

### 快速验证

```bash
# 只验证代码质量
/sdlc validate

# 只验证需求实现
/sdlc verify

# 只运行测试
/sdlc test

# 只做安全检查
/sdlc secure
```

## 关键改进

1. **分离 validate 和 verify**:
   - `validate`: 代码质量 (机器检查)
   - `verify`: 需求验证 (对比 spec)

2. **质量保证流程**:
   - validate → verify → test → secure → cr
   - 每步有明确的目的和检查内容

3. **自动化程度**:
   - validate: 高度自动化
   - verify: 半自动化 (需读取 spec)
   - test: 高度自动化
   - secure: 高度自动化
   - cr: 人工为主，工具辅助
